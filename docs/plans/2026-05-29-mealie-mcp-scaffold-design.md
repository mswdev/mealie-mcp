# Mealie MCP ‚Äî Scaffold Design

**Date:** 2026-05-29
**Branch:** feature/scaffold ‚Üí develop
**Scope:** First PR ‚Äî project scaffold only. No Mealie tools beyond one proof-of-concept `get_about` tool.

---

## 1. Goal

Establish the full project skeleton for a public TypeScript MCP server that exposes Mealie's complete REST API to any MCP-compatible client. This PR does not implement tools ‚Äî it sets up the toolchain, conventions, and structure so every subsequent PR can move fast.

**Problems with existing Mealie MCPs:** incomplete API coverage, unmaintained. This project solves both by generating types directly from Mealie's OpenAPI spec and building to the full spec from the start.

---

## 2. Architecture

### Runtime Flow

```
MCP Client (Claude Desktop, Claude Code, Cursor, ChatGPT, etc.)
        ‚îÇ
        ‚îÇ  stdio (JSON-RPC) ‚Äî default local use
        ‚îÇ  OR HTTP Streamable ‚Äî for hosted/remote use
        ‚ñº
  src/index.ts          ‚Üê process entry; selects transport from TRANSPORT env var
  src/server.ts         ‚Üê McpServer instance; registers all tool groups
  src/config.ts         ‚Üê zod-validated env vars; exits with clear error if invalid
  src/logger.ts         ‚Üê pino instance; writes to stderr only (stdout = JSON-RPC)
  src/client/
    MealieClient.ts     ‚Üê typed fetch wrapper using generated types
  src/types/
    mealie.ts           ‚Üê generated by openapi-typescript from Mealie's OpenAPI spec
  src/tools/
    about.ts            ‚Üê scaffold proof-of-concept tool (get_about)
```

### Transport Strategy

The server supports two transports, selected via the `TRANSPORT` environment variable:

| Mode | Transport | Use Case |
|------|-----------|----------|
| `stdio` (default) | `StdioServerTransport` | Local installs ‚Äî Claude Desktop, Claude Code, Cursor, VS Code, Zed |
| `http` | `StreamableHTTPServerTransport` | Hosted/remote deployments ‚Äî ChatGPT, Codex, any HTTP MCP client |

The same tool code runs on both transports. Only `index.ts` differs between modes.

### Key Constraint

Pino **must** log to `stderr` exclusively. `stdout` is reserved for JSON-RPC in stdio mode; writing anything else to `stdout` corrupts the transport.

## 4. Configuration

Environment variables validated by zod at startup. The server exits with a descriptive error if required vars are missing.

| Variable | Required | Default | Description |
|----------|----------|---------|-------------|
| `MEALIE_URL` | Yes | ‚Äî | Base URL of the user's Mealie instance (e.g. `https://mealie.example.com`) |
| `MEALIE_API_TOKEN` | Yes | ‚Äî | Mealie API token (generate in Mealie ‚Üí Profile ‚Üí API Tokens) |
| `TRANSPORT` | No | `stdio` | `stdio` or `http` |
| `PORT` | No | `3000` | HTTP port; only used when `TRANSPORT=http` |

---

## 5. Type Generation Pipeline

Mealie ships an OpenAPI spec at `<MEALIE_URL>/openapi.json`. The `generate-types` script pulls this spec and runs `openapi-typescript` to produce `src/types/mealie.ts`.

- Generated types are **committed to the repo** ‚Äî users cloning the project get types immediately without running generation.
- Run `npm run generate` to regenerate against a live Mealie instance (e.g. when Mealie releases a new version).
- The default generation target is the public Mealie demo instance; override with `MEALIE_SPEC_URL` env var.

## 6. Build & Distribution

- **Bundler:** `tsup` (esbuild-based) ‚Äî produces a single `dist/index.js` (ESM, Node 18+)
- **Shebang:** `#!/usr/bin/env node` prepended to output by tsup
- **Distributed via npm** ‚Äî users run via `npx` with no global install

### MCP Client Config (stdio)

```json
{
  "mcpServers": {
    "mealie": {
      "command": "npx",
      "args": ["-y", "mealie-mcp"],
      "env": {
        "MEALIE_URL": "https://your-mealie-instance.com",
        "MEALIE_API_TOKEN": "your-token-here"
      }
    }
  }
}
```

### Hosted HTTP Mode

```bash
TRANSPORT=http PORT=3000 MEALIE_URL=https://... MEALIE_API_TOKEN=... npx mealie-mcp
```

## 7. Toolchain

| Tool | Purpose |
|------|---------|
| `@modelcontextprotocol/sdk` | MCP server core (tools, transports, schema) |
| `zod` | Tool input schema validation + env var validation |
| `pino` | Structured logging to stderr |
| `pino-pretty` (dev) | Human-readable log output during development |
| `openapi-typescript` | Generates TypeScript types from Mealie's OpenAPI spec |
| `tsup` | Bundles TypeScript to single ESM output file |
| `@biomejs/biome` | Lint + format (replaces ESLint + Prettier) |
| `vitest` | Unit tests |
| Native `fetch` | HTTP client for Mealie API (Node 18+ built-in, no extra dep) |

## 8. Testing Approach

- `vitest` for unit tests, colocated with source files (`about.test.ts` next to `about.ts`)
- Scaffold includes one test: `src/tools/about.test.ts` ‚Äî verifies the `get_about` tool formats its response correctly using a hand-written fake for `MealieClient`
- No integration tests in the scaffold ‚Äî those come in subsequent tool PRs
- Quality gate: `npm run build && npm run test && npm run lint` must pass before any commit
